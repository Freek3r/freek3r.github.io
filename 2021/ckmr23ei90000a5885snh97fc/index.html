XSS——跨站脚本攻击

什么是XSS

XSS 是攻击者在Web页面中插入恶意的脚本代码，当用户浏览该网页时，嵌入其中的脚本代码就会被执行，从而达到恶意攻击用户的目的。

XSS产生原因

XSS 漏洞产生的原因实际上就是对输出内容未进行安全的过滤或转义而导致用户浏览器解析了嵌入其中的恶意脚本造成的。

XSS的类型

  1. 反射型——Reflection

反射性XSS 只是简单的把用户的输入不经过处理或处理不完全就返回给浏览器而导致的。也就是说，反射性XSS 需要诱导用户点击一个恶意的链接才能成功。这种攻击方式多与 社会工程学 相结合。

  1. 储存型XSS——Stored

储存型XSS 会把拥护的输入储存在服务端中，每次访问某个页面时都会从服务端数据库取出数据并返回给用户。如果服务端不在用户输入数据时或给用户发送数据时进行过滤，那么就会导致用户浏览器解析恶意脚本，从而达到攻击目的

这种攻击方式可以蠕虫化。例如微信朋友圈的评论里存在 XSS 漏洞，那么攻击者可以进行构造一个恶意脚本，使每个看到评论的用户自动给其他好友朋友网进行评论并插入恶意代码，从而造成大范围的用户被攻击。

  1. DOM型XSS——DOM Based

这种类型的 XSS 并非按照“数据是否保存在服务器端”来划分的，从效果上来说也是 反射型XSS 单独划分出来的，因为 DOM型XSS 的形成原因比较特别，是由于客户端脚本自身解析不正确导致的安全问题。

XSS防护

  1. 对重要的 Cookies 使用 HTTPOnly 参数
  2. 检查所有用户可控的输入点。对用户的输入进行严格的检查、过滤和拦截。
  3. 检查所有的输出点。因为 XSS 最终的攻击点是在输出点，所以需要对输出的信息进行过滤和转义等
  4. 正确处理富文本（注意区分 富文本 与 XSS）。在文章、论坛等需要用到富文本的地方，严格禁止所有的危险标签及事件，原则上应当使用白名单过滤标签、事件及属性。

CSRF——跨转请求伪造

什么是CSRF

CSRF（Cross-Site Request Forgery），即跨站请求伪造攻击。尽管听起来像 XSS ，但它与 XSS 非常不同，XSS 利用站点内的信任用户，而 CSRF 则通过伪装成受信任用户的请求来利用受信任的网站。与 XSS攻击相比， CSRF 攻击往往不大流行（因此对其进行防范的资源也相当稀少）和难以防范。

CSRF产生原因

在用户访问完A网站并未进行退出登录操作后，去浏览了B页面，而B页面早已被攻击者嵌入了恶意的请求代码。所以攻击者从而可以在B网站以你的名义往A网站发送恶意请求，对A网站的服务器来说这个请求是完全合法的，但是却完成了攻击者所期望的一个操作，比如以你的名义发送邮件、发消息，盗取你的账号，添加系统管理员，甚至于购买商品、虚拟货币转账等。

CSRF 简单的利用方法

上面这个只是简单的利用模板，URL 要填 CRSF 漏洞所在的链接， METHOD 填请求方法，而 INPUTS 要根据需要传的参数构造 input 标签。 CSRF 漏洞的利用还是需要日积月累的经验和完美的社工来完成。

CSRF防护

  1. 来源校验：使用 HTTP 请求头中 Referer 来源，对客户端源进行身份校验，此方法早期使用比较多，但是仍然容易被绕过，所以这里并不建议使用。
  2. 用户 token 校验：添加基于当前用户身份的有效 token 随机验证机制，即在向后端提交数据操作请求时，需要带上当前的随机 token 进行校验，此种校验方法当前使用比较多
  3. 添加验证机制：在请求数据时，填写或交互验证码进行人机验证，从而防止恶意未授权操作
